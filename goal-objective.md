# ZIAAP Review-Candidate Goal

## Primary objective

Develop ZIAAP into a visually excellent, conceptually complete, internally
truthful, and self-explanatory synthetic showcase of:

> AI-native dispute governance and arbitration

The present goal is achieved when the complete review-ready prototype has been
built, internally verified, documented, and packaged for external evaluation.

External reviewer sessions and feedback-driven revisions belong to the subsequent
validation and improvement cycle. External validation must remain visibly marked
as pending. Never fabricate reviewer participation, findings, usability results,
comprehension results, or independent validation.

The prototype remains synthetic and simulation-only. Production arbitration
infrastructure, confidential dispute processing, legally effective appointments,
production signatures, operative awards, sovereign runtime implementation,
Council-of-LLMs orchestration, DLT execution, and smart-contract enforcement are
outside this goal.

## Required sequence

```text
Decision-complete foundation and alpha plan
→ integrity and authority foundation
→ opening experience and Party Alignment
→ Arbitral Reasoning Calibration
→ Stress Testing
→ Protocol Manifest and Simulated Appointment
→ Later Synthetic Dispute
→ final demonstration dossier
→ complete documentation synchronization
→ internal technical, usability, and quality verification
→ reviewer package
→ freeze prototype-showcase-v1.0-review-candidate
→ external reviewer sessions in the next cycle
```

The foundation, authority model, opening experience, Party Alignment alpha,
attack tests, documentation checkpoint A, and initial reviewer materials already
exist in the current worktree and must be preserved and improved rather than
discarded.

## Authority and status model

Keep the existing orthogonal dimensions:

```ts
lifecycleMode: "simulation_only" | "authoritative"

executionStatus:
  | "not_executed"
  | "illustrative_only"
  | "executed_unverified"
  | "validated"
  | "failed"

lifecycleStatus:
  | "draft"
  | "manifest_prepared"
  | "manifest_acknowledged"
  | "appointment_simulated"
  | "dispute_simulated"
  | "closed"

legalEffect: false
```

Only `simulation_only` is available. `authoritative` and `validated` are reserved
and unreachable. The review-ready prototype may create only:

- `not_executed`;
- `illustrative_only`;
- `executed_unverified`; and
- `failed`.

Illustrative and executed-unverified artifacts may be acknowledged for the
simulated ceremony. Failed and not-executed artifacts may not. No artifact may
receive authoritative approval.

Every consequential artifact must show lifecycle mode, execution status, actor,
authority, provenance, version, consequence, legal-effect boundary, synthetic
status, no production identity verification, no institutional appointment, no
production signature, and no legally operative award.

## Integrity requirements

Preserve the completed integrity safeguards:

- separate **View illustrative example** and **Run live execution** actions;
- illustrative fixtures return `illustrative_only`;
- successful live calls return `executed_unverified`;
- live errors fail closed and never return fixture success;
- failures preserve prior valid artifacts but block acknowledgement until
  deliberate recovery;
- model output cannot assign itself an authoritative validation verdict;
- synchronous predicates control UI readiness only;
- the asynchronous simulated-appointment command rebuilds and hashes the Exact
  protocol manifest immediately before mutation and rejects stale, inconsistent,
  forged, incomplete, or bypassed state without mutation; and
- the public showcase disables unrestricted live execution and retrieval.

Use **Exact protocol manifest**. Explain that it identifies selected contractual
decisions, legal architecture, Constitution, declared model/prompt/retrieval/tool/
engine versions, stress artifacts, fictional human-arbitrator record, and change
policy. It is not a build/dependency/provider/environment/runtime attestation or
production cryptographic signature.

## Stage 1: Party Alignment

Preserve the complete demonstrated chain:

```text
Draft contract
→ independent party expectations
→ divergence analysis
→ source-linked legal and commercial issues
→ options and trade-offs
→ deterministic scenario testing
→ revised contractual language
→ exact-version bilateral confirmation
→ Alignment Annex
```

Source text, party assertion, AI-supported analysis, legal constraint,
deterministic calculation, proposed option, and agreed contractual text remain
visually distinct.

## Stage 2: Arbitral Reasoning Calibration

- evidence;
- fairness;
- commercial values;
- remedy boundaries;
- discretion;
- uncertainty;
- escalation;
- model and tool policy;

State clearly that model weights remain unchanged. Make calibration visibly
different from stress testing.

## Stage 3: Stress Testing

For each hypothetical show:

- facts;
- evidence;
- required safeguards;
- illustrative or live output;
- execution status;
- limitations; and
- bilateral acknowledgement for simulated ceremony.

Do not describe illustrative or executed-unverified output as validated.

## Stage 4: Protocol Manifest and Simulated Appointment

Show:

- the full manifest contents;
- manifest version;
- SHA-256 digest;
- exact selected artifact references;
- bilateral simulated acknowledgement;
- fictional arbitrator disclosure;
- simulated arbitrator acceptance; and
- simulation-only/no-effect status.

Use:

> Simulated appointment under the acknowledged protocol manifest

The appointment transition must use the integrity-enforcing async command and
must never create ordinary authoritative `appointed` status.

## Stage 5: Later Synthetic Dispute

Show:

```text
Synthetic dispute arises
→ acknowledged protocol manifest is identified
→ shared merits record is assembled
→ optional sealed settlement facilitation
→ provisional simulation-only determination
→ fictional human arbitrator independently reviews
→ simulated human decision
```

Settlement content remains separate from merits reasoning. The protocol
determination remains provisional. The final simulated human decision remains
non-operative and without legal effect.

## Definition of done

The present goal is complete when:

- Stages 1–5 form a coherent end-to-end synthetic demonstration;
- the opening explains ZIAAP without presenter intervention;
- Party Alignment is fully demonstrated;
- calibration and stress testing are visibly distinct;
- the Exact protocol manifest and simulated appointment flow work;
- the later synthetic dispute flow works;
- the fictional human arbitrator remains visibly in command;
- settlement information remains separated from merits reasoning;
- the complete 12-artifact dossier is available;
- all consequential artifacts show the required authority metadata;
- README and canonical documentation are synchronized;
- type-check, lint, tests, production build, and critical desktop/mobile paths
  pass;
- blocked, loading, failure, reset, dossier, and invalid-transition states work;
- the reviewer package is complete;
- unresolved external-validation questions are documented; and
- the repository is frozen as `prototype-showcase-v1.0-review-candidate`.

External reviewer sessions begin only after this review candidate is frozen.

## Operating rules

- Inspect the repository before each major phase.
- Keep architecture proportional to a showcase, not a production platform.
- Use only synthetic, domain-neutral examples in public code and documentation.
- Update code, tests, fixtures, and canonical documentation together.
- Use small, reviewable commits.
- Preserve the distinction between Party Alignment, reasoning calibration,
  stress testing, independent validation, manifest acknowledgement, simulated
  appointment, and human legal authority.
- Record assumptions, decisions, unresolved risks, and verification results.
- Continue autonomously until the review candidate is frozen or a genuine
  technical blocker requires user input.
- Never claim legal, institutional, security, usability, comprehension, or
  validation properties that the prototype does not establish.
